# FHIR → OMOP Learning Project

A structured, hands-on learning environment for mastering **FHIR R4 → OMOP CDM**
transformation on a Databricks lakehouse, with a specific focus on **Indian ABDM profiles**.

**Designed for:** Data engineers with Databricks/Delta Lake experience who are new to
clinical data standards (FHIR, OMOP, ABDM).

> **Note:** This is a learning project, not a production codebase. Code is written to be
> readable and well-commented rather than optimised. The pipeline runs on Databricks
> Community Edition (free tier) against synthetic patient data — no real PHI is involved.

---

## What's in this repo

| Layer | What it does |
|-------|-------------|
| **Bronze** | Land raw Synthea FHIR bundles (JSON) into Delta Lake as-is |
| **Silver** | Flatten each FHIR resource type into typed Delta tables |
| **Gold** | Transform Silver into OMOP CDM v5.4 tables |
| **dbt** | Silver → Gold transformations with incremental merge strategy |
| **Tests** | pytest unit tests runnable locally without a cluster |
| **Docs** | Mapping decisions log, ABDM gap analysis, learning log |

The synthetic patient population is generated by [Synthea](https://github.com/synthetichealth/synthea)
with a Maharashtra (India) demographic module to approximate Indian patient data.

## Prerequisites

- **Databricks Community Edition** (free) — [sign up](https://community.cloud.databricks.com/)
- **Java 11+** (for Synthea)
- **Python 3.9+** (for local tests)
- **dbt-databricks** (for Gold transformations)

No AWS account or paid Databricks workspace is required to follow along. The notebooks
use Unity Catalog Volumes instead of S3 for file storage.

## Quick Start

### 1. Clone and set up

```bash
git clone https://github.com/<your-username>/fhir-omop-project
cd fhir-omop-project

pip install pytest pyspark fhir.resources
pytest tests/ -v   # runs locally, no cluster needed
```

### 2. Generate synthetic FHIR data

```bash
git clone https://github.com/synthetichealth/synthea ../synthea
cd ../synthea
./run_synthea -p 100 "Maharashtra" --exporter.fhir.export=true
# Copy bundles to your Databricks Volume (see notebook 01 for instructions)
```

### 3. Run the pipeline on Databricks

Import the notebooks in order:

```
notebooks/01_synthea_setup.py       ← upload FHIR files to a Unity Catalog Volume
notebooks/02_bronze_ingestion.py    ← JSON bundles → raw_bundles Delta table
notebooks/03_silver_flattening.py   ← raw_bundles → typed per-resource tables
```

Each notebook has a detailed markdown header explaining what it does, what to expect
in the output, and any Databricks-specific gotchas.

## Key Mapping Decisions

See [`docs/mapping_decisions.md`](docs/mapping_decisions.md) for the full log. Highlights:

- **`_lineage_id`** — `SHA-256(source_file | fhir_resource_id)` is used as a stable,
  deterministic merge key across all three layers (Bronze → Silver → Gold).
- **ABHA ID extraction** — uses `FILTER + get()` instead of `identifier[0]` to safely
  handle patients without an ABDM identifier (expected for Synthea data).
- **UDF for bundle explode** — FHIR `entry[].resource` is polymorphic; `from_json` can't
  handle a variable schema, so a Python UDF re-serialises each resource to a JSON string.

Useful prompts:

## Using with Claude (claude.ai)

1. Create a new **Claude Project** at claude.ai
2. Paste the contents of `CLAUDE_PROJECT_SYSTEM_PROMPT.md` into Project Instructions
3. Attach as Project files:
   - `docs/mapping_decisions.md`
   - `docs/abdm_gap_analysis.md`
   - ABDM FHIR IG PDF (download from abdm.gov.in)
   - OMOP CDM v5.4 DDL (from OHDSI GitHub)

Every conversation in that Project will then have full context of the stack, learning
progress, and mapping decisions.
